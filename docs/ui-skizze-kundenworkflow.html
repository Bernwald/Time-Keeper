Implementierungsplan

Reihenfolge nach kritischem Pfad. Jede Phase ist eigenständig deploybar und liefert Mehrwert — keine „Big Bang"-Migration.

  1. Phase 1Architektur-Doktrin schriftlich fixieren
    docs/architektur.html ergänzen: „Sources & Chunks als first-class Objekte". Gemeinsame Sprache zwischen dir, Mitgründer und künftigen Entwicklern.
  2. Phase 2Schema-Migration
    Neue Migration YYYYMMDDHHMMSS_sources_and_chunk_versions.sql: Tabelle sources (type, external_id, etag, last_synced_at, sync_status), Tabelle chunk_versions (append-only mit model_version, valid_from/valid_to), Soft-Delete + Tombstones für DSGVO. RLS pro organization_id.
  3. Phase 3aSharePoint-Connector (read-only)
    Edge Function supabase/functions/connector-sharepoint/. OAuth via Microsoft Graph, Initial-Sync legt Files in raw_events, bestehender worker-normalize verarbeitet weiter. Delta-Sync per /delta-Endpoint, Cron alle 5 Min. Beweist die Sync-Hypothese.
  4. Phase 4aChat-UI mit Citations + Drilldown
    apps/web/src/app/(app)/chat/, Citations als wiederverwendbares Component mit Highlight-Mapping (chunk → byte_range im Original). Mobile-first ab 360px, Drilldown-Panel klappt unter 880px nach unten. Beweist die UX-Hypothese.
  5. Phase 3bWriteback-Connector
    Edge Function connector-writeback/: Edit/Delete aus UI → PATCH/DELETE gegen Graph API. Bei Erfolg: betroffene Chunks invalidieren, neue Version anhängen. Löst das eigentliche Kundenproblem.
  6. Phase 4bQuellen- & Wissens-UI
    (app)/quellen/ und (app)/wissen/ mit Server Actions für Writeback-Calls.
  7. Phase 5Admin-KPIs
    (admin)/quellen-health/: Sync-Status pro Connector, Fehler-Logs, Coverage-KPIs. Sichtbarkeit bleibt Berater-Entscheidung.
  8. Phase 6Zweiter Connector (Google Drive)
    Erst wenn SharePoint stabil läuft. connector-gdrive/ nutzt dasselbe Interface, derselbe Worker-Pfad.
