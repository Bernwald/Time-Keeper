1 · Überblick

Der Google-Drive-Connector holt periodisch Dateien aus dem verknüpften Drive-Account, legt sie als raw_events ab, normalisiert sie, erzeugt Embeddings und stellt den Inhalt in der Chat-/RAG-Suche bereit. Das Ganze läuft als mehrstufige Queue-Pipeline auf Supabase Edge Functions und pgmq, angetrieben durch pg_cron.

Pipeline-Stufen:

1
connector-gdrive
Edge Function · holt Dateien aus Drive API
Walkt Drive-Baum, lädt Bytes/Text, schreibt in raw_events, enqueued in normalize.
2
worker-normalize
pgmq-Worker · alle 30 s
Liest aus normalize-Queue, ruft upsert_connector_source(...) und enqueued in embed.
3
worker-embed
pgmq-Worker · alle 30 s
Chunkt Text, ruft OpenAI-Embeddings, schreibt content_chunks, flippt sources.status = 'ready' + sync_status = 'success'.
4
RAG-Search
Chat + Telefonassistent
Vector-Similarity-Search über content_chunks.embedding.

2 · Automatische Trigger & Timings

WasZeitplanImplementiert alsZweck
Delta-Sync (gdrive) alle ~5 min via pg_cron cron.schedule('gdrive-delta-...') → invoke_edge_function('connector-gdrive', {"action":"delta-sync"}) Holt Drive-Änderungen seit letztem page_token (Changes API).
Reconcile (gdrive) täglich 17 3 * * * (03:17 UTC) cron.schedule('reconcile-connectors-daily', ...) Full-Listing, entfernt verwaiste Sources (soft-delete).
worker-normalize alle 30 s cron.schedule('worker-normalize-30s', '30 seconds', ...) Drainiert pgmq.q_normalize, ruft DB-RPC upsert.
worker-embed alle 30 s cron.schedule('worker-embed-30s', '30 seconds', ...) Drainiert pgmq.q_embed, erzeugt Embeddings.
Full-Resync + Reconcile manuell UI-Button „Aufräumen" → reconcileConnector() in apps/web/src/app/quellen/actions.ts Initial-Sync + Reconcile sequentiell.
Initial-Sync manuell oder nach OAuth-Verknüpfung Button „Jetzt synchronisieren" → triggerInitialSync() Vollständiger Tree-Walk (nicht Changes-API), setzt frischen page_token.
Erneut verarbeiten manuell pro Datei RetryButton → retrySource() Setzt sync_status='queued', enqueued normalize-Msg mit letztem raw_event.

Queue-Durchsatz: batch 25 Messages pro Worker-Tick, visibility_timeout 60 s, max 5 Retries pro Message, danach dead-letter in job_failures.

3 · Dateidiscovery (Drive API)

Zwei Modi — abhängig davon, ob bereits ein page_token in organization_integrations.config gespeichert ist.

3.1 Initial-Sync (kein page_token)

Rekursiver Tree-Walk — pflichtweise, weil files.list mit corpora=user keine Dateien aus geteilten Ordnern zurückgibt, wenn der User nur über Ordnervererbung Zugriff hat (z. B. jemand hat in deinem Timekeeper-Ordner ein Dokument hochgeladen).

Ablauf in listAllFilesRecursive():

  1. Seed 1: alle Ordner des Users ('me' in owners) — als folder-Query.
  2. Seed 2: alle „Mit mir geteilten" Ordner (sharedWithMe = true).
  3. Seed 3: eigene + mit mir geteilte Files (nicht-Ordner) direkt einsammeln.
  4. BFS: für jeden gefundenen Ordner '<folderId>' in parents abfragen — sammelt auch Dateien ein, die anderen gehören aber im geteilten Ordner liegen.
  5. Dedupliziert per Map<id, DriveFile>.
  6. Nach Walk: frischen startPageToken holen und in config.page_token speichern.

Query-Parameter in jedem Call:

q=trashed = false and mimeType != 'application/vnd.google-apps.folder'
supportsAllDrives=true
includeItemsFromAllDrives=true
pageSize=1000

Datei: supabase/functions/_shared/google-drive.ts

3.2 Delta-Sync (mit page_token)

Nutzt drive/v3/changes mit dem gespeicherten page_token. Die Changes API liefert automatisch alles, was sich seit dem Token geändert hat. Ordner werden im Code ausgefiltert (die Changes API akzeptiert keinen q-Filter).

Caveat: Drive's Changes API sieht manchmal Dateien in geteilten Unterordnern nicht sofort. Deshalb gibt es den täglichen Reconcile + die manuelle „Aufräumen"-Aktion mit Full-Resync.

4 · Parser & Textextraktion

Der Drive-Connector versucht für jede Datei Text zu extrahieren und legt das Ergebnis als _extracted_text im raw_events-Payload ab. Der embed-Worker nutzt diesen Text direkt — ohne eigene Extraktion.

TypMIMEParserErgebnis
Google Docs vnd.google-apps.document Drive Export text/plain voll
Google Sheets vnd.google-apps.spreadsheet Drive Export text/csv voll
Google Slides vnd.google-apps.presentation Drive Export text/plain voll
Word (.docx) ...wordprocessingml.document JSZip → word/document.xml → XML-Strip voll
Excel (.xlsx) ...spreadsheetml.sheet JSZip → xl/sharedStrings.xml + xl/worksheets/sheet*.xml voll
PowerPoint (.pptx) ...presentationml.presentation JSZip → ppt/slides/slide*.xml voll
HTML / Markdown / JSON / CSV / XML / Plain Text text/*, application/json, … alt=media + response.text() voll
Google Forms vnd.google-apps.form — Stub (nur Metadaten)
PDF application/pdf — noch nicht implementiert Stub
Bilder / Video / Audio image/*, video/*, audio/* — Stub (kein OCR/Transkript)
Ordner vnd.google-apps.folder — werden im Connector gefiltert ignored

OOXML-Extraktion (docx/xlsx/pptx): JSZip wird per import("https://esm.sh/jszip@3.10.1") lazy geladen. Das Script strippt XML-Tags, behandelt Entities (&amp;, &lt; …) und komprimiert Whitespace. Siehe Funktionen extractOoxmlText() und stripXml() in google-drive.ts.

5 · Pipeline raw_events → normalize → embed

A
connector-gdrive schreibt raw_events
supabase/functions/connector-gdrive/index.ts
Für jede gefundene Datei: Download-Text (sofern unterstützt), Payload sanitizen (Null-Bytes stripen, weil Postgres JSONB sie ablehnt), einmal in raw_events inserten (idempotent per payload_hash), dann Message in pgmq.q_normalize.
B
worker-normalize → upsert_connector_source
supabase/functions/worker-normalize/index.ts
Liest Payload aus raw_events, ruft PG-Function upsert_connector_source(...):
  • Neue Source → Insert mit sync_status='queued'.
  • Bestehende Source, Etag geändert → Update + chunks invalidieren + enqueue embed.
  • Bestehende Source, Etag gleich, aber sync_status war queued/failed/error → Force-Retry: chunks invalidieren + enqueue embed.
  • Bestehende Source mit deleted_at → wird immer aus dem Papierkorb geholt.
Bei was_changed = true enqueued der Worker eine Message in pgmq.q_embed mit Titel, extrahiertem Text und source_id.
C
worker-embed → OpenAI + content_chunks
supabase/functions/worker-embed/index.ts
Chunkt den Text (Ziel ~400 Tokens, 50 Overlap), ruft sequentiell embedText() (OpenAI text-embedding-3-small), schreibt content_chunks (alte soft-deleted). Erst nach erfolgreichem Schreiben flippt sources.status = 'ready' und sync_status = 'success'.

6 · Status-Modell

Eine Source hat zwei Status-Spalten, die zusammen das UI-Verhalten steuern:

sources.status

Wird in der /sources-Liste („Dateien") angezeigt.

  • ready — fertig, in RAG verfügbar
  • processing / pending — in Arbeit
  • error — fehlgeschlagen

sources.sync_status

Wird in der /quellen-Seite (Connector-Übersicht) angezeigt.

  • success → UI-Label „indexiert"
  • queued → „wartet"
  • processing → „verarbeitet"
  • error / failed → „Fehler"
  • idle — Default
Wichtig: Beide Felder werden vom worker-embed gemeinsam auf ready / success gesetzt — aber erst nachdem die Embeddings tatsächlich in content_chunks liegen. Wenn du in /quellen „indexiert" siehst, existieren die Embeddings garantiert.

7 · Fehler & Retry

StelleVerhalten
connector-gdrive bei Drive-API-Fehler Wirft Exception → runSync setzt Run auf failed, schreibt in job_failures.
connector-gdrive bei Download-Fehler einer Einzeldatei console.warn, Datei wird trotzdem als Stub ingested (Text = null).
raw_events Insert (z. B. Null-Byte in Payload) Eintrag in job_failures, andere Files laufen normal durch. Payload wird bereits im Connector gesanitized.
worker-normalize Exception Kein ACK, visibility_timeout (60s) läuft ab, Message wird erneut gelesen. Nach 5 Versuchen → dead-letter in job_failures.
worker-embed / OpenAI-Fehler embedText() wirft mit echtem Status + Body → Retry via visibility_timeout. Nach 5 Versuchen: sources.sync_status = 'error' + dead-letter.
OpenAI-Key fehlt Wird als explizite Exception erkannt und landet in job_failures.error_message. Secret muss in Supabase Edge Function Secrets gesetzt sein (OPENAI_RESEARCH_TIMEKEEPER_KEY oder OPENAI_API_KEY).

8 · Manuelle Aktionen in der UI

Button / AktionServer-ActionEffekt
„Jetzt synchronisieren" triggerInitialSync('google_drive') POST connector-gdrive mit action: initial-sync. Löscht config.page_token nicht — aber das Backend nutzt bei mode=initial sowieso einen frischen Tree-Walk.
„Aufräumen" reconcileConnector('google_drive') Zuerst POST initial-sync, dann POST reconcile. Sequentiell. Zieht neue Dateien + entfernt verwaiste.
„Erneut verarbeiten" (pro Zeile) retrySource(sourceId) Setzt sync_status='queued', nimmt das letzte raw_event und enqueued eine neue normalize-Message. Der Force-Retry-Branch in upsert_connector_source sorgt dafür, dass auch bei unverändertem Etag re-embedded wird.
„Löschen" (pro Zeile) deleteSource(id) Soft-Delete via soft_delete_source(id), verschwindet aus /sources und /quellen.
„Wiederherstellen" (im /papierkorb) restoreSource(id) Clear deleted_at, setzt sync_status='queued'. Nächster Sync re-embedded.
„Endgültig löschen" (im /papierkorb) purgeSource(id) Hard-Delete der Source + content_chunks. Point of no return.

11 · Troubleshooting-Rezepte

Eine Datei taucht nach Sync nicht in /sources auf

  1. Ist sie in Drive sichtbar, wenn du als der verknüpfte User einloggst? Evtl. nur mit einzelnen Files geteilt, nicht in einem eigenen Ordner.
  2. Check raw_events:
    select * from raw_events where provider_id='google_drive' and payload->'file'->>'name' ilike '%NAME%'
  3. Wenn drin: check sources + sync_status. Falls queued → worker-normalize/embed klemmt.
  4. Wenn nicht drin: check job_failures nach Unicode-Errors o. Ä.
  5. Letzte Option: „Aufräumen" drücken — macht Full-Resync + Reconcile in einem.

Status bleibt ewig auf „wartet"

  1. Queue-Länge prüfen:
    select queue_name, queue_length from pgmq.metrics_all() where queue_name in ('normalize','embed');
  2. Recent Worker-Logs: Supabase Dashboard → Edge Functions → worker-embed / worker-normalize.
  3. select * from job_failures order by failed_at desc limit 10; — zeigt den echten Fehler.
  4. Typische Ursachen: OpenAI-Key fehlt, Rate-Limit, Netzwerkfehler zu Drive/OpenAI.

Dateien aus einem geteilten Ordner fehlen

Der rekursive Walk holt nur Ordner ein, die entweder dem User gehören oder explizit mit ihm geteilt wurden (sharedWithMe). Wenn die Datei in einem Ordner liegt, auf den nur ein Link-Share existiert, wird sie nicht gefunden. Lösung: Ordner explizit mit dem verknüpften Account teilen.

Suche / Telefonassistent findet nichts

  1. select count(*) from content_chunks where embedding is not null; — 0? Dann Embed-Pipeline steckt.
  2. Pro Source:
    select title, word_count, (select count(*) from content_chunks c where c.source_id=s.id and c.embedding is not null) as emb from sources s where connector_type='gdrive';
  3. Wenn word_count klein (6-7) → Parser hat nichts extrahiert, nur Stub angelegt. Datei-Typ prüfen.

Bei Änderungswünschen: Dieses Dokument zeigt, welche Datei zu welchem Bereich gehört. Für einen guten Claude-Prompt: beschreib das beobachtete Verhalten, den erwarteten Zustand, und nenn die Datei/Section aus dieser Doku als Kontext — dann spart sich Claude das Raten.
